# 提交代码

## 特性

- 如果没有暂存，则自动暂存变更文件
- 建议按照不同的关注点及特性拆分提交
- 使用中文，避免使用 emoji 表情，考虑可读性内容精简，突出重点。
- 忽略对 i18n 文件（如 zh-ch.json）、lock 文件(pnpm-lock.yaml) 的变更分析，但需要提交这部分变更
- 禁止在 master 分支提交代码

## Commit Types

- feat: 新特性
- fix: 问题修复
- docs: 文档或注释变更
- refactor: 没有改变逻辑的代码重构
- style: 例如代码格式化，补充标点等
- perf: 性能优化
- test: 添加或调整测试用例
- chore: 工具、配置变更

## 实践

### 好的实践

- **提交前确认分支与暂存**：先确认不在 master，再看 `git status` 与暂存内容；未暂存时先分析变更再 `git add`，避免误提交
- **按类型与原子性拆分**：一次提交只做一件事；无法用一句话说清时就拆成多个 commit；合并 master 产生的变更可忽略不单独分析
- **消息格式**：类型 + 简短说明 + 必要原因。例：`fix: 修复画布缩放时选区错位，因未按当前 zoom 换算坐标`
- **命令式、说明动机**：用「添加」「修复」「移除」等现在时；解释为什么而不只写做了什么。例：`refactor: 将请求逻辑抽到 useRequest，便于复用和单测`
- **pre-commit 失败**：不代为修复，由我选择如何处理

### 坏的实践

- **在 master 上提交**：必须先切到功能分支再 commit
- **一条 commit 混合多类变更**：功能、修 bug、格式等混在一起 → 拆成多个 commit
- **消息模糊或无信息量**：`fix: 修复 bug`、`update`、`修改代码` → 看不出改了什么、为何改
- **使用 emoji 或英文混写**：`✨ feat: add button` → 用中文、不用 emoji
- **误提交不该提交的**：临时调试、本地配置、未检查的 i18n 整文件等 → 提交前检查 `git status` 与 diff
